Ingestion adapters all produce DNSEvent objects, so the rest of the pipeline is source-agnostic. Start with file-based log ingestion (easiest to test), then add live packet capture.

Key libraries: dnspython, scapy, pyshark, watchdog (for log tailing).

Phase 2 — Feature engineering (week 2–3)
This is the intellectual core of the project. The features you extract determine everything the detectors can see.

Entropy is the single most important signal. High-entropy subdomains are the hallmark of base64/hex-encoded exfil data:

python
import math
from collections import Counter

def shannon_entropy(s: str) -> float:
    counts = Counter(s.lower())
    length = len(s)
    return -sum((c/length) * math.log2(c/length) for c in counts.values())

# legitimate: "mail.google.com"          → entropy ≈ 3.1
# tunneling:  "a3f9b2c1d4e5.evil.com"   → entropy ≈ 4.6
Additional features per query: subdomain length, label count, longest label, consonant-to-vowel ratio (DGA detection), unique characters, digit ratio, _ and - counts.

Per-host over a time window: query rate, unique FQDNs, NXDomain ratio, answer-to-query ratio, bytes exfiltrated estimate.

Phase 3 — Detection engines (week 3–5)
Build these in order of complexity — rule-based first, ML second.

Statistical engine — threshold rules on computed features. Start here; these catch the obvious tools immediately and give you labeled data for ML training.

Temporal engine — beaconing detection. Compute inter-arrival times for queries from the same source to the same domain family. C2 beacons have surprisingly consistent timing; calculate mean and standard deviation, flag sources where the coefficient of variation is below 0.1.

ML classifier — train on labeled data (iodine/dnscat2 = positive, your own browsing traffic = negative). scikit-learn's RandomForestClassifier is a strong baseline; add IsolationForest for anomaly detection on unlabeled traffic.

Reputation engine — integrate free threat intel feeds (Cisco Umbrella top 1M as allowlist, abuse.ch DNS blocklist as denylist). Implement DGA detection using character-level n-gram models trained on known-good vs known-DGA domains.

Phase 4 — Scoring and output (week 5–6)
The scoring engine takes signals from all four detectors and produces a composite risk score:

python
WEIGHTS = {
    "entropy_score":    0.30,
    "temporal_score":   0.25,
    "ml_score":         0.30,
    "reputation_score": 0.15,
}

def composite_score(signals: dict) -> float:
    return sum(WEIGHTS[k] * v for k, v in signals.items())
Alert thresholds: > 0.4 = low, > 0.6 = medium, > 0.8 = high. Tune these against your lab traffic.

What you just built — and what each file does:

ingest.py reads a .pcap or plain log file and emits DNSEvent objects. The pcap parser is pure stdlib — no scapy, no dpkt. It handles Ethernet and raw IP linklayers.

features.py computes two classes of features: stateless per-query signals (entropy, subdomain length, query type, digit ratio) and stateful per-host signals tracked over a sliding time window (query rate, NXDOMAIN ratio).

detect.py scores each event against six weighted rules, prints colour-coded findings, and renders a ranked summary of offending hosts.

What the test run showed: the iodine-style base64 exfil traffic scored 0.55–0.63 (MEDIUM) and was caught immediately. The beaconing and DGA traffic landed at 0.23–0.39 — visible at --min-score 0.2 and clearly separated from clean hosts, but below the default threshold so they don't create noise in normal operation. That's intentional — those are the signals to improve next.
